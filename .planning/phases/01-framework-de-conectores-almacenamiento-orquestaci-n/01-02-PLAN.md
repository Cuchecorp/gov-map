## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| fetcher â†” URL de la cola | el worker no debe fetchear URLs arbitrarias (SSRF); solo hosts de fuentes conocidas |
| Edge runtime â†” R2 | la firma S3 (SigV4) debe ser correcta y los secrets no loguearse |
| ingesta â†” esquema cambiante | el crudo debe capturarse aunque el esquema cambie (drift se registra, no rechaza) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering | SSRF / abuso del fetcher con URL no confiable | mitigate | El framework expone un hook de allow-list de hosts; el DummyConnector declara su host explicitamente; el worker (Plan 03) valida host antes de fetch. En M1 sin fuentes reales, el riesgo es bajo pero el hook queda en su sitio |
| T-01-05 | Spoofing/Tampering | firma S3 mal implementada para R2 | mitigate | aws4fetch (SigV4 probado con SubtleCrypto), NUNCA hand-roll de SigV4 |
| T-01-06 | Information Disclosure | secrets R2 logueados en errores del fetcher/r2-store | mitigate | No incluir headers de auth en mensajes de error/log; tests verifican que el error de 412/!ok no expone credenciales |
| T-01-07 | Tampering | crudo sobrescrito en R2 (race check-then-put) | mitigate | content-addressing (key=sha) + `If-None-Match: *` atomico; 412 = idempotente OK (FND-02 inmutabilidad) |
| T-01-08 | Denial of Service | drift estricto rechaza crudo y pierde la captura | mitigate | drift solo registra `drift_alert` y continua; zod en ingesta es shape-guard suave, no estricto (validacion estricta es Fase 5+) |
</threat_model>
