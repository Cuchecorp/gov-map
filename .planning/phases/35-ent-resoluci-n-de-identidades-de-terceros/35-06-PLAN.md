<objective>
Forward-fix the broken FK in resolver_entidad (Issue 1 from 35-05-SUMMARY.md, BLOCKER ENT-03/ENT-04).

Migration 0036 inserts into identidad_audit with vinculo_id = v_vinculo_id where v_vinculo_id comes from
vinculo_entidad (the TERCEROS link table). But identidad_audit.vinculo_id has FK → vinculo_identidad (the
PARLIAMENTARY link table). Against PROD where vinculo_identidad is empty, the first confirm-with-promote
throws identidad_audit_vinculo_id_fkey (SQLSTATE 23503). 0036 is applied to PROD and IMMUTABLE.

The fix (Option A, operator-locked) is a NEW forward-fix migration 0037 that:
  1. Adds identidad_audit.vinculo_entidad_id bigint references vinculo_entidad(id)
  2. Adds CHECK identidad_audit_un_solo_vinculo: num_nonnulls(vinculo_id, vinculo_entidad_id) <= 1
     (note: <= 1, not = 1 — the reject/no-promote path writes audit rows with BOTH columns null,
      num_nonnulls=0, which must remain legal)
  3. CREATE OR REPLACE resolver_entidad — body identical to 0036 EXCEPT the audit INSERT writes
     vinculo_entidad_id (value v_vinculo_id) instead of vinculo_id
  4. Defensive deny-by-default grant block emitted AFTER the function definition

What closes the 35-05 Issue-1 bug:
  (a) The corrected RPC INSERT writing vinculo_entidad_id — the terceros id no longer lands in
      the parliamentary FK column.
  (b) The FK on identidad_audit.vinculo_entidad_id — structural guard on the correct column.
  The CHECK (num_nonnulls <= 1) is defense-in-depth against both-columns-set simultaneously,
  NOT the guard against the FK-violation class.

Purpose: Close ENT-03/ENT-04 (human confirmation path for terceros). Makes ENT-01 schema-correct
on PROD. ENT-01 pgTAP closure (0035 green) is owned by 35-07 running in parallel.
