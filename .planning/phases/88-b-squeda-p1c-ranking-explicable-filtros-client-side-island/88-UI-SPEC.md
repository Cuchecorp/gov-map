A client-only island (`app/components/buscar-filtros.tsx`) that filters/reorders the already-fetched result slice IN MEMORY (zero network, zero Supabase — FichaRail contract), plus a server-side pure state normalizer (`app/lib/estado-bucket.ts`). ## Interaction Contract (phase-specific — consumed by executor + checker)

### Filter panel layout
- Desktop (≥`md`/768px): filter controls above or beside the result list, using the existing `.net-filtros`-style grouping (flex-wrap, `gap` groups, `bg-muted` container, `border`, `rounded-[var(--radius)]`). No Supabase; pure React state (`useState`, per `AutoresList`/`red-graph` island precedent).
- Mobile 390px: filter groups stack vertically (`flex-direction: column`, reduced gaps) exactly like `.net-filtros` `@media (max-width: 47.99rem)`. All facet chips remain operable; none removed or hidden on mobile.

### Facets rendered (FILT-01)
Year · Iniciativa (Mensaje/Moción) · Estado (buckets) · Cámara de origen. **Partido facet is NOT rendered** in this phase (LOCKED — arrives with BIO-03/P2). The island accepts an optional `partido` field on each slice row so P2 can populate it without refactor; absent → the facet group simply does not render (no disabled placeholder that confuses).

### Chips + counts (FILT-02)
- Each facet value is a chip showing its label + honest count "de estos N resultados" (e.g. `En tramitación · 7`). The count reflects the currently-fetched result set ONLY — a visible legend states this explicitly: **"Conteos sobre estos N resultados, no sobre todo el corpus."** Never presented as a global corpus count.
- Facets with count 0 are **disabled** (`aria-disabled="true"`, `disabled` on the button, reduced opacity `opacity-0.4` per `.net-b-pager__btn:disabled`), never removed — the citizen sees which buckets exist but are empty here.
- NULL/unmapped estado renders as an explicit **`sin dato`** bucket chip (visible, neutral slate), never silently folded into a substantive bucket.

### Estado normalizer (FILT-03)
- Server-side pure lib `app/lib/estado-bucket.ts`: free text (`proyecto.estado`/`etapa`) → enum bucket. Table-driven, exported, reusable (ficha de proyecto and /agenda may consume later — do not couple to the island).
- Buckets: `en_tramitacion`, `publicado_ley`, `archivado`, `rechazado`, `retirado`, `sin_dato`. Final substantive set derived from DISTINCT real PROD values (read-only during dev) + tests with real corpus examples — no invented categories.
- Unmapped/NULL → `sin_dato` (honest explicit bucket), NEVER silently to a substantive bucket. Runs server-side; the island receives the bucket already computed on each slice row.
- Bucket display labels (Spanish, must pass anti-insinuación linter): En tramitación · Publicado / Ley · Archivado · Rechazado · Retirado · Sin dato.

### Year facet (honest source)
- `ProyectoRow` has NO filing-date column (only `fecha_captura` = capture date, which is NOT the project's year and must NOT be shown as such). The server component derives the year for each row from the earliest tramitación event ("Ingreso" `fecha`) when enriching the slice; when no filing date is derivable → the row's year is `null` and falls into a `Sin dato` year bucket (same honesty rule as estado). Never derive "year" from `fecha_captura`. Never derive year from the boletín suffix (it is the materia code, not the year).

### Order toggle + ranking (RANK-01)
- A visible order toggle with 3 deterministic, explained options: **Relevancia (por defecto) · Más recientes · Mensajes primero**. Implemented as a segmented control / toggle group; the engaged option carries `aria-pressed="true"` + petróleo active treatment.
- Primary order is ALWAYS the retrieval rank (relevance, already from the RPC). The explainable ranking acts as tie-break and as opt-in reorder:
  - Relevance ties → Mensaje (Ejecutivo) before Moción, then more recent before older.
  - "Más recientes" → sort by derived year desc (rows with `Sin dato` year sink to the end, never fabricated to a date).
  - "Mensajes primero" → Mensaje group before Moción group, relevance-order within each.
- A visible, honest rules legend (must pass linter): **"Orden por relevancia de la búsqueda; en empates se priorizan los mensajes del Ejecutivo y lo más reciente."** No parlamentario score, no "polémica", nothing inferred.
- Per-result explanatory labels: a neutral `Mensaje`/`Moción` chip + year chip on each result card so the *why* of the order is legible. Reuse the outline Badge; NO score, NO cosine, NO rank number ever surfaced (existing `SearchResultCard` §5 rule).

### Accessibility (LOCKED)
- Fully keyboard operable: every chip/toggle is a real `<button type="button">` (not a div), tab-reachable, Enter/Space activatable.
- `aria-pressed` on facet chips and order toggles reflecting engaged state (mirror existing island `aria-current`/`aria-pressed` precedent in `search-box.tsx` / `red-graph.tsx`).
- `focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-accent-product` on all controls.
- Disabled facets: `disabled` + `aria-disabled="true"`; not focus-trapping.
- 44px (`min-h-11`) touch targets on all interactive controls (mobile 390px).
