**Extensibility mandate (CONTEXT LOCKED — third multi-dataset section):**
`/parlamentario/[id]` is a stack of independent `<section>`s. Phase 10 shipped `#votos`,
Phase 11 appended `#lobby`. This phase appends `#patrimonio`. **Each section is its own Suspense
boundary, its own `<h2>`, its own honest empty/error state, and its own provenance lane.** No
section composes a datum from another block (a vote, a lobby meeting, a declaration) into a single
highlighted unit. Patrimonio sits in its `mt-12` lane and never visually pairs with votos or lobby.

**Phase-11 lesson, made structural (CRITICAL):** in Phase 11 the `representado` field rendered
product-authored connective prose ("en representación de {X}") that was NOT in the §10 payload
contract and NOT exercised by the content-gate test — it slipped past the gate. This phase forbids
that pattern by construction: **every field rendered — in the row list AND in the comparison view —
must be in the §10 payload contract, prefixed with a muted noun label (never a product-authored
verb/preposition that pairs entities), and exercised by the §9.1 content-gate test.** The comparison
view is the highest-risk surface for this and is covered explicitly in §3.5 and §9.1.

## 3. Patrimonio Section Layout

### 3.0 Placement in the shell (LOCKED — separate lane)

```
<main className="max-w-3xl mx-auto px-4 md:px-8 py-8 md:py-16">
  <Suspense fallback={<ParlamentarioHeaderSkeleton />}>
    <ParlamentarioHeader id={id} />                  {/* Phase 10 §3.1 — reused verbatim */}
  </Suspense>

  <section id="votos"     className="mt-12"> … </section>   {/* Phase 10 — untouched */}
  <section id="lobby"     className="mt-12"> … </section>   {/* Phase 11 — untouched */}

  <section id="patrimonio" className="mt-12">               {/* THIS PHASE */}
    <h2 className="text-xl font-semibold mb-4">Declaraciones de patrimonio e intereses</h2>
    <Suspense fallback={<PatrimonioSkeleton />}>
      <PatrimonioSection id={id} searchParams={...} />      {/* §3.2–3.5 */}
    </Suspense>
  </section>
</main>
```

- **`mt-12` gutter is the lane boundary.** The patrimonio `<h2>` is a sibling of `#votos`/`#lobby`
  with 48px breaks. A declaration is NEVER rendered inside the same `<article>`/`<Card>`/`<li>`/`<tr>`
  as a vote or a lobby meeting. Structural anti-insinuation rule (§9.1).
- **Own Suspense boundary** so a slow probidad query never blocks votos/lobby and the section renders
  its own honest empty/error state independently.
- **`PatrimonioSection` is a Server Component.** Reads the declaration tables via the anon key; RLS
  public-read only on the non-sensitive published columns; any RUT, familiar data, or internal key is
  deny-by-default and never selected into the public payload.

### 3.1 Section intro line + CC BY 4.0 attribution (mandatory, the honest framing)

Directly under the `<h2>`, before the version history, one muted line of context (text-sm,
muted-foreground) PLUS a persistent CC BY 4.0 attribution line:

```
"Declaraciones de patrimonio e intereses presentadas ante el Consejo para la Transparencia
 (InfoProbidad). Cada versión se muestra tal como la declaró el parlamentario, con su fecha de
 presentación."

"Fuente: InfoProbidad — Consejo para la Transparencia. Datos bajo licencia CC BY 4.0."  [↗]
```

- This frames every datum as: *what the legislator declared, on this presentation date, as the source
  publishes it* — never as a verified statement of current net worth, never as an audit.
- **CC BY 4.0 attribution is mandatory and VISIBLE in this intro AND repeated in the comparison view
  (§3.5)** — CONTEXT LOCKED: the attribution must appear in derived views, not only the primary list.
  Render it as plain text + an external link to the license (`creativecommons.org/licenses/by/4.0`)
  and/or the InfoProbidad source. Never hide it behind a tooltip-only affordance.

### 3.2 Version history — `DeclaracionVersionRow` (the freshness spine)

Reverse-chronological list of declaration versions for this legislator, ordered by
`fecha_presentacion DESC` (most recent first). Each version is one row. **The presentation date is
the most prominent element of the row** — an old declaration must NEVER read as current state.

```
<ul>  (declaration versions; one <li> per version)
  <li>  (DeclaracionVersionRow)
    [ fecha de presentación — font-mono, PROMINENT (text-base), + amber freshness badge if stale ]
    [ tipo de declaración: "Patrimonio" / "Intereses" — text-sm muted, literal de la fuente ]
    [ resumen literal de campos (key/value) OR "Ver detalle" disclosure — §3.3 ]
    [ ProvenanceBadge ]                          ← ml-auto, per version, mandatory
  </li>
</ul>
```

**`DeclaracionVersionRow` anatomy (LOCKED rules):**

- **Fecha de presentación — the prominence rule (INT-04, LOCKED):** `font-mono`, format `DD MMM YYYY`
  (es-CL), rendered at `text-base` (NOT the muted `text-sm` used for secondary metadata) so it is the
  visual anchor of the row. It is **labeled** "Presentada el {fecha}" so it can never be misread as
  "current". If the declaration is stale relative to "now" (reuse `esStale` / the inherited 48h-style
  threshold, OR — preferred for declarations — a domain threshold such as "older than the current
  legislative period"), the `ProvenanceBadge` renders its **amber freshness** state AND the row carries
  the standing caveat copy (§6.4): an old declaration is explicitly marked as historical, never as the
  present state.
- **Tipo de declaración:** literal source label ("Declaración de patrimonio" / "Declaración de
  intereses"), `text-sm`, muted. Never editorialized.
- **Field values — LITERAL ONLY:** every declared field is rendered verbatim from the source
  (parsed with zod when structured; extracted literally via the `@obs/fichas` golden-gate path when
  the source is a PDF/free text, and ONLY after passing the `data-routing` PII gate — no RUT/PII to any
  LLM). **No computed field, no normalization that implies judgment, no rounding that changes meaning,
  no LLM summary of "what this means".** A monetary value is shown as the source states it; if the
  source gives a range or a category, show that category verbatim. Each field carries a **muted noun
  label** (e.g. "Bienes inmuebles:", "Actividades:", "Valores:") — NEVER a product-authored verb or
  preposition that pairs entities (this is the `representado` lesson, §9.1 rule below).
- **ProvenanceBadge:** per version, mandatory. `sourceName = "InfoProbidad"`, `capturedAt =
  fecha_captura`, `sourceUrl =` the per-declaration source record link. Unknown provenance →
  `sourceName="fuente desconocida"`, no link (§6.3). Never omit the badge. The amber state is the
  freshness signal called for in CONTEXT.

**No CamaraChip per row.** Chamber/role lives in the header; repeating it per declaration row risks
compositing identity context into the declaration datum. Keep the row about: presented-when, what was
declared (literal), per which source.

**Pagination:** if a legislator has many versions, paginate server-driven via `?patrimonioPage=N`
(page size **10 versions**), "Anteriores" / "Siguientes" anchors (`min-h-[44px]`) and "Página N de M"
— identical pattern to Phase 10/11. SSR so a journalist can deep-link a version page. No infinite scroll.

### 3.3 Field disclosure (per version)

Each version's declared fields render as a sober key/value block (`<dl>` with `<dt>` muted noun label
+ `<dd>` literal value). For a long declaration, the fields collapse behind a server-driven
"Ver detalle de la declaración" disclosure (anchor `?ver=<versionId>`, `min-h-[44px]`) so the version
list stays scannable. **No client accordion required** — server-driven expansion keeps it RSC and
deep-linkable. The disclosed detail is the same literal `<dl>`; never a richer "analyzed" view.

### 3.4 Identity guard — declaration ↔ legislator (LOCKED, INT-05, riesgo #1)

The declaration attaches to THIS profile ONLY when the declarant matched `determinista`/`confirmado`
via `correrPipeline`, fixed through the typed `EnlaceConfirmado` invariant (Phase 9). Each
adjudication leaves a row in `identidad_audit`.

- A `confirmado` declaration → renders as a normal `DeclaracionVersionRow`, attributed to this legislator.
- A `probable`/`revision`/`no_confirmado` declaration → FK is NULL; **never silently attributed** and
  **never aggregated** into any version count or comparison for this legislator. If surfaced at all it
  lives in a clearly-labeled "menciones sin verificar" area with the raw declarant mention +
  `<IdentityMarker />` — but the default is to exclude it from this profile.
- Because this is **PII**, the cross-link is even stricter than lobby: only deterministic/confirmed RUT
  or fully-confirmed identity attaches a declaration here. A NULL link never degrades to "probably this
  person".

### 3.5 Side-by-side comparison — `DeclaracionComparacion` (LOCKED — SOLO DATOS, the strongest rule)

The citizen can compare two (or more) declaration versions over time, **showing only the data,
side by side**. This is the highest-risk surface in the phase and the one where the Phase 11
`representado` lesson must not repeat.

```
[ "Comparar versiones" — server-driven selector: pick two versions by fecha de presentación ]
   selector = anchor links/checkboxes → ?comparar=<versA>,<versB>  (SSR, min-h-[44px])

<Table>  (shadcn Table — sober data grid)
  <caption> "Comparación de declaraciones. Cada columna es una versión, fechada por su fecha de
             presentación. Se muestran solo los datos declarados, sin cálculo ni interpretación."
             + CC BY 4.0 attribution line (repeated here per CONTEXT)
  <thead>
    <tr> [ Campo ] [ Presentada el {fechaA} ] [ Presentada el {fechaB} ] </tr>   ← fecha = font-mono, prominent
  </thead>
  <tbody>
    <tr> [ noun-label campo ] [ valor literal A ] [ valor literal B ] </tr>   ← one row per declared field
    …
  </tbody>
</Table>

[ ProvenanceBadge for version A ]   [ ProvenanceBadge for version B ]   ← one per column, mandatory
```

**LOCKED rules for the comparison view (the §9.1 gate covers THIS view, not only the list):**

1. **ZERO verdict column. ZERO computed delta.** There is NO "diferencia", NO "variación", NO
   "aumentó/disminuyó", NO "Δ", NO "enriquecimiento", NO "conflicto de interés", NO percentage change,
   NO any cell the product computes by combining two declared values into a judgment. The table holds
   ONLY the literal declared values, each in its own dated column. If a field is absent in one version,
   the cell reads "No declarado en esta versión" (literal fact), never "—" implying a drop, never a
   highlighted gap.
2. **No valence coloring. No "diff" styling.** This is a sober data table, NOT a git-diff. No red/green
   cells, no strikethrough, no "added/removed" badges, no arrows. A changed value looks exactly like an
   unchanged value — same neutral text, same color. The reader compares the two columns themselves; the
   product never marks one as better/worse/larger/smaller. (Color contract §5 forbids any valence hue.)
3. **Every field rendered is in the §10 payload contract** and carries a muted **noun** label — never a
   product-authored connecting verb/preposition that pairs entities (no "pasó de … a …", no
   "en representación de", no "que aumentó a"). This is the structural fix for the Phase 11 `representado`
   gap: a field that is not in the contract and not gate-tested must not appear in the table.
4. **Presentation date is the column identity** and is rendered prominently (`font-mono`, labeled
   "Presentada el") so the comparison is unambiguously "version-at-date vs version-at-date", never
   "before vs after" framing that implies a timeline of cause.
5. **Provenance + CC BY 4.0 per column**, mandatory and visible — the attribution is repeated in this
   derived view (CONTEXT LOCKED).
6. **Identity guard applies** — only `confirmado` versions are selectable for comparison; unverified
   declarations never enter the grid.
7. **Comparison is SSR** (`?comparar=A,B`) — deep-linkable, RSC, no client diff engine.

### 3.6 NO aggregate verdict / NO viz / NO cross-composition (anti-feature, LOCKED)

Explicitly PROHIBITED this phase (FEATURES.md INT anti-features + PROJECT.md hard anti-feature
"enrichment/conflict verdict" + CONTEXT lane rule):

- **No "enrichment" anything** — no net-worth trend line, no growth chart, no "patrimonio en el tiempo"
  sparkline, no total/sum the product computes across versions, no ranking against other legislators.
- **No conflict-of-interest flag/badge/score**, no "posible conflicto", no linking a declared interest
  to a vote, a bill, a lobby meeting, a contract, or a donation. **No co-occurrence claim** ("declaró
  interés en X y votó Y").
- **No CSS bar / no chart.** Unlike Phase 10 attendance, there is NO honest aggregate to plot here; do
  not invent one. (`VotacionBar` exists but is not used.)
- **No composition across lanes:** a declaration row/cell must not embed, reference, or sit inside a
  vote/lobby/contract/donation unit, and vice versa. Cross-block adjacency is deferred (CONTEXT
  "Deferred Ideas") and gated by legal review (Phase 13+).

## 6. States

### 6.1 Empty / honest-empty states (LOCKED — never reads as "no assets" or "clean")

The existential distinction: **"no hemos ingerido" ≠ "no tiene declaraciones" ≠ "declaró cero".**
A legislator with zero ingested declarations must NOT look identical to one with a fully-loaded
record. Three honest states, mirroring Phase 10 §3.6 / Phase 11 §6.1:

| Scope | Trigger | Copy |
|-------|---------|------|
| Patrimonio section — not yet ingested | confirmed-declaration query returns 0 AND ingestion not yet run for this legislator | "Aún no hemos ingerido las declaraciones de patrimonio e intereses de este parlamentario. Esto no significa que no haya declarado — los datos de InfoProbidad se están incorporando." |
| Patrimonio section — ingested, zero confirmed | ingestion ran, 0 confirmed declarations for this legislator | "No se registran declaraciones de patrimonio e intereses confirmadas para este parlamentario, según InfoProbidad." |
| Declarant mention unverified | source lists a declaration whose declarant link is not `confirmado` | excluded from this profile by default; if surfaced, "Menciones sin verificar" area with raw mention + `<IdentityMarker />` — never attributed, never compared |
| Comparison — fewer than 2 confirmed versions | only one (or zero) confirmed version exists | comparison selector hidden; muted line "Se necesita más de una versión para comparar. Hay {N} versión registrada." (fact, not a deficiency) |
| Field absent in one compared version | a field exists in version A but not B | cell reads "No declarado en esta versión" (literal); never "—", never a highlighted/colored gap |

Copy principles (v1.0 §6.1 / Phase 10 / Phase 11, hardened): state the fact, give context, never say
"error" for missing data, never "Oops", **never let an empty read as virtue, wealth, poverty, honesty,
or absence of holdings.** Always distinguish "no ingestado todavía" from "ingestado, sin resultados".

### 6.2 Loading states

shadcn `Skeleton`, shape-matched: `PatrimonioSkeleton` mirrors the real structure — intro+attribution
line skeleton + 3 version-row skeletons (prominent fecha bar + tipo bar + field-block bar + provenance
bar). One skeleton per Suspense boundary; container `aria-hidden="true"` (no focus). Pattern identical
to Phase 10 `VotosSkeleton` / Phase 11 `LobbySkeleton`. The comparison table, when active, shows a
table-shaped skeleton (header row + N body-row bars).

### 6.3 Error states

Inherited from v1.0 §6.3 / Phase 10 / Phase 11. A Supabase/network failure renders inside
`border border-destructive/20 bg-destructive/5 rounded-lg p-4 text-sm`:

"Ocurrió un error al cargar las declaraciones de patrimonio e intereses. Si el problema persiste,
puedes consultar la fuente directamente en [InfoProbidad ↗]." + `<button>Reintentar</button>`
(`text-primary underline`).

- Distinguish a real DB/network error (throw → honest error UI) from "no rows" (→ §6.1 empty). Use
  checked errors / `.maybeSingle()` discipline — never mask a transient error as "sin declaraciones".
- **Degraded / quarantined ingestion (CONTEXT — drift is BLOCKING for this PII source):** if
  InfoProbidad is unreachable or the run is quarantined for drift, the section renders ONLY honest
  states — it NEVER fabricates declarations and NEVER shows a partial declaration as complete. If a row
  is fixture-sourced + `human_verification`, its ProvenanceBadge reflects that honestly (source/date as
  captured), never a fake official link. Drift here quarantines the run rather than degrading silently.
- **Provenance unavailable:** render ProvenanceBadge with `sourceName="fuente desconocida"`, no link.
  Never hide the badge — its absence falsely implies the datum has no source.

### 6.4 Freshness / "old declaration" standing caveat (LOCKED — INT-04)

When a version is stale (older than the freshness threshold), in addition to the amber ProvenanceBadge,
the row carries a muted standing caveat so an old declaration is never read as current:

"Esta es una declaración histórica, presentada el {fecha}. No representa necesariamente el estado actual."

This is the analogue of the Phase 10 rebeldías method footnote — a standing caveat that defuses
misreading. It is plain text, never colored beyond the inherited amber of the badge itself.

## 7. Interaction & Responsive

| Breakpoint | Layout |
|-----------|--------|
| Mobile `< 640px` | Single column, `px-4`. Version rows stack: fecha (prominent) + tipo on one line, field `<dl>` below, ProvenanceBadge wraps to its own line. **Comparison table** becomes horizontally scrollable within its container (`overflow-x-auto`), OR stacks to a per-field two-row layout (label, then the dated values) — never truncated. |
| Tablet `640–1024px` | Single column, `px-6`. Comparison table fits two version columns. |
| Desktop `> 1024px` | `max-w-3xl mx-auto px-8`. Single column — no sidebar. ProvenanceBadge `ml-auto` per row. Comparison table may use the full section width with `overflow-x-auto` if more than two versions are compared. |

**Interactions (all SSR/server-driven):**

## 8. Accessibility

Inherited verbatim from v1.0 §8 / Phase 10 / Phase 11. Non-negotiable:

- **No information by color alone** — no declaration value or comparison cell conveys meaning via hue;
  freshness is amber AND carries the "declaración histórica, presentada el…" text; identity uncertainty
  is the `IdentityMarker` text, not a color. A changed value in the comparison table is NOT colored.
- **Comparison table semantics:** a real `<table>` with `<caption>`, `<th scope="col">` for the dated
  version columns and `<th scope="row">` for each field label, so a screen reader announces
  "{campo}, Presentada el {fechaA}: {valorA}; Presentada el {fechaB}: {valorB}". The caption carries the
  "solo datos, sin cálculo" framing and the CC BY 4.0 attribution.
- **Presentation date prominence is also semantic** — labeled "Presentada el {fecha}" in text, not by
  size alone, so AT users get the historical framing.
- **IdentityMarker:** `⚠` is `aria-hidden`; the span `aria-label="identidad no verificada"`. Text always
  exactly "identidad no verificada". (Phase 11 WARNING noted the inherited `title`-only rationale is not
  AT-reliable — out of scope to patch here; flagged for the shared component.)
- **Heading hierarchy:** the shell stays valid — `h1` = nombre (one per page); `h2` = "Votaciones",
  "Reuniones de lobby", "Declaraciones de patrimonio e intereses" as siblings; `h3` = "Comparar
  versiones" / "Menciones sin verificar" only if rendered. No skipped levels.
- **List semantics:** versions are a `<ul>`/`<li>` list (each `<li>` one version), readable in order.
- **Field key/value:** `<dl>`/`<dt>`/`<dd>` so a screen reader pairs label and literal value.
- **Contrast:** WCAG AA. Civic tokens at L=38% meet it on white.
- **Keyboard:** pagination, disclosure, comparison selector, and all links are real anchors; focus rings
  preserved (shadcn `ring`).
- `lang="es"` on `<html>` (inherited).

### 9.1 Tone rules — HARD CONTENT RULE: NO verdict + anti-insinuation lane + anti-causality (riesgo existencial #2)

These are enforced at the component and copy level and are the most important constraint of the phase.
A checker/auditor failure on any of these BLOCKS the phase. They EXTEND Phase 10 §9.1 and Phase 11 §9.1
(both remain in force) and — critically — **the content gate covers the comparison view AND every field
label, not only the row list** (the Phase 11 `representado` lesson).

1. **ZERO verdict — the strongest rule of this phase (LOCKED, PROJECT.md hard anti-feature).** The
   product NEVER computes or implies "enriquecimiento", "conflicto de interés", "incremento patrimonial",
   "variación", "delta", "aumentó/disminuyó", a percentage change, or any judgment derived from combining
   declared values. The comparison shows literal values in dated columns and nothing else. PROHIBITED
   verbatim (gate regex must cover): "enriqueci", "conflicto de inter", "aument", "disminuy", "incrementó",
   "variaci", "delta", "Δ", "creció", "pasó de", "más rico", "patrimonio total" (as a product-computed sum).
2. **No causality, ever.** Never assert or imply *why* something was declared or what followed. No
   "declaró para", "a cambio de", "antes de votar", "que resultó en".
3. **No relationship/affinity language and no product-authored connective prose that pairs entities
   (the `representado` rule).** Every field is a muted NOUN label + literal value. PROHIBITED: any
   product-authored verb/preposition joining two named values ("en representación de", "vinculado a",
   "asociado con", "pasó de X a Y"). If a relation is *in the source data*, it is shown as a literal
   field value with a noun label and IS in the §10 contract and IS gate-tested — never as authored prose.
4. **No score / index / ranking / flag.** No wealth ranking, no conflict score, no "patrimonio elevado"
   flag, no comparison-to-peers. Literal values only.
5. **No judgment adjectives** about a declaration, a value, or the legislator ("elevado", "sospechoso",
   "millonario", "modesto", "opaco", "incompleto" beyond the literal source state).
6. **Freshness honesty (INT-04).** An old declaration is explicitly labeled historical (amber +
   standing caveat §6.4); it NEVER reads as the current state.
7. **Identity uncertainty** uses exactly "identidad no verificada" — never a hedge.
8. **Third-party / familiar privacy is absolute (LEGAL-03).** Never expose the legislator's RUT, a
   familiar's name/RUT, or any internal-only field; these are deny-by-default and never selected into the
   public payload, never shown, never in a tooltip/`title`. Only what InfoProbidad already publishes is shown.
9. **Provenance + CC BY 4.0 are mandatory** on every version AND repeated in the comparison view; if
   provenance unknown, "fuente desconocida" — never omitted. Attribution never hidden behind tooltip-only.
10. **An empty is a fact, not a virtue.** Never phrase absence of declarations (or of a field) as
    cleanliness, wealth, poverty, integrity, or "no tiene patrimonio". "No ingestado" and "ingestado, sin
    declaraciones" are distinct.

**Gate-test mandate (closes the Phase 11 gap):** the content-gate test for this section MUST exercise
BOTH the version list AND the comparison view (including a populated multi-version comparison with a
changed value and an absent field), and MUST assert that (a) no field renders outside the §10 contract,
(b) no product-authored connective verb/preposition appears, (c) no verdict/delta vocabulary (rule 1
regex) appears in any cell, caption, or label, (d) no RUT/familiar pattern appears in `textContent`,
(e) the CC BY 4.0 attribution string is present in the comparison caption.

## 10. Provenance & Identity Data Flow (UI contract with backend)

Every declaration version carries:

```ts
interface DeclaracionVersionRow {
  declaracion_id: string;
  version_id: string;
  tipo: "patrimonio" | "intereses";        // literal source category → labeled, not editorialized
  fecha_presentacion: string;              // ISO; PROMINENT, → "Presentada el DD MMM YYYY"
  // legislator (declarant) identity guard — §3.4
  parlamentario_id: string | null;         // set ONLY if confirmado
  parlamentario_estado_vinculo: "confirmado" | "probable" | "revision" | "no_confirmado" | null;
  parlamentario_mencion: string;           // raw declarant name from source (unverified case only)
  // declared content — LITERAL ONLY, each field a noun-labeled key/value
  campos: Array<{
    etiqueta: string;                      // muted NOUN label (e.g. "Bienes inmuebles") — NEVER a verb/prep
    valor: string;                         // literal value verbatim from source (incl. "No declarado en esta versión")
  }>;
  // NOTE: legislator RUT, familiar names/RUTs, and any internal key are NEVER part of the public payload
  //       (LEGAL-03 deny-by-default; explicit anon revoke on the new PII tables per CONTEXT)
  origen: string;                          // → "InfoProbidad" → ProvenanceBadge.sourceName
  fecha_captura: string;                   // → ProvenanceBadge.capturedAt (drives amber freshness via esStale)
  enlace: string | null;                   // → ProvenanceBadge.sourceUrl (per-declaration source record)
  licencia: "CC BY 4.0";                   // → visible attribution (intro + comparison caption)
  es_historica: boolean;                   // freshness flag → amber badge + standing caveat §6.4
}
```

- The `@obs/probidad` connector populates provenance + the CC BY 4.0 license per version and sets the
  declarant identity link via the typed `EnlaceConfirmado` invariant (Phase 9): the FK is set ONLY on
  `determinista`/`confirmado`; otherwise NULL + raw mention. Each decision leaves a row in `identidad_audit`.
- **Only `confirmado` versions attach to this profile**, enter the version count, and are selectable for
  comparison; `probable`/`revision`/`no_confirmado` are excluded by default and never compared (§3.4).
- **`campos` are LITERAL** — structured source parsed with zod (no LLM); PDF/free-text extracted via the
  `@obs/fichas` golden-gate path ONLY after the `data-routing` PII gate (no RUT/PII to any LLM). The UI
  renders `campos[].valor` verbatim with `campos[].etiqueta` as a muted noun label. The UI computes NOTHING
  across versions — the comparison table is a pure side-by-side of `campos` from each selected version.
- **RLS contract:** public-read on the non-sensitive published columns above; legislator RUT, any familiar
  data, internal stable keys, and any column flagged sensitive are deny-by-default for `anon` (with explicit
  `revoke` per CONTEXT/LEGAL-03) and MUST NOT be selected into this payload. Data-routing LLM gate applies.
- **Drift is BLOCKING (CONTEXT):** a partial/anomalous run quarantines rather than emitting silent rows; the
  UI therefore only ever shows complete, attributed versions or an honest empty/error state.
