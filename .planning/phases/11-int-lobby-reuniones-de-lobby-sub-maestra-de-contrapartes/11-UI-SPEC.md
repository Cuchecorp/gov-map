## 3. Lobby Section Layout

### 3.0 Placement in the shell (LOCKED — separate lane)

```
<main className="max-w-3xl mx-auto px-4 md:px-8 py-8 md:py-16">
  <Suspense fallback={<ParlamentarioHeaderSkeleton />}>
    <ParlamentarioHeader id={id} />               {/* Phase 10 §3.1 — reused verbatim */}
  </Suspense>

  <section id="votos" className="mt-12"> … </section>   {/* Phase 10 — untouched */}

  <section id="lobby" className="mt-12">                 {/* THIS PHASE */}
    <h2 className="text-xl font-semibold mb-4">Reuniones de lobby</h2>
    <Suspense fallback={<LobbySkeleton />}>
      <LobbySection id={id} searchParams={...} />        {/* §3.2 */}
    </Suspense>
  </section>

  {/* Phase 12+ append <section id="patrimonio">, … here. Same rule: own <h2>,
      own Suspense, own empty state, own provenance lane. */}
</main>
```

- **`mt-12` gutter is the lane boundary.** The lobby `<h2>` and the votos `<h2>` are siblings at
  the same level with a 48px break between them. A lobby meeting and a vote are NEVER rendered
  inside the same `<article>`, `<Card>`, or list item. This is the structural form of the
  anti-insinuation rule (§9.1).
- **Own Suspense boundary** so a slow lobby query never blocks votos, and so the section can
  render its own honest empty/error state independently.
- **`LobbySection` is a Server Component.** Reads `lobby_audiencia` + `lobby_contraparte` via the
  anon key; RLS public-read on the non-sensitive columns; sensitive columns (any RUT of a third
  party, internal keys) deny-by-default and never selected for the public view.

### 3.1 Section intro line (the honest framing — mandatory)

Directly under the `<h2>`, before the list, one muted line of context (text-sm, muted-foreground):

```
"Audiencias registradas bajo la Ley del Lobby (Ley 20.730). Cada reunión se muestra tal como
 la publica leylobby.gob.cl."
```

This sets the reader's frame BEFORE any row: these are *published audiences*, the counterpart is
*as the source lists it*, and nothing here asserts a relationship beyond "se reunieron, en esta
fecha, según esta fuente". It is the analogue of the rebeldías method footnote in Phase 10 — a
standing caveat that defuses misreading.

### 3.2 Lobby meeting list — `LobbyAudienciaRow`

Reverse-chronological list of audiences for this legislator (sujeto pasivo), ordered by
`fecha DESC`. Each row is one audience. The row is a sibling of `VotoFichaRow` in structure and
discipline, but it carries lobby data only — it never references a vote.

```
<ul>  (lobby audiences; one <li> per audience)
  <li>  (LobbyAudienciaRow)
    [ fecha — font-mono text-sm text-muted-foreground ]
    [ contraparte: TEXTO CRUDO  (+ IdentityMarker only if NOT confirmed) ]
    [ asunto / materia de la reunión — text-sm, literal de la fuente, opcional ]
    [ ProvenanceBadge ]                          ← ml-auto, per row, mandatory
  </li>
</ul>
```

**`LobbyAudienciaRow` anatomy (LOCKED rules):**

- **Fecha:** `font-mono`, format `DD MMM YYYY` (es-CL), e.g. "14 may 2026". The date of the
  audience as published.
- **Contraparte (the lobbyist / firm) — TEXTO CRUDO, the central anti-insinuation rule:**
  - Render the counterpart string **exactly as the source publishes it** (`contraparte_nombre`).
  - Link it to a `lobby_contraparte` / identity profile **ONLY if the match is `confirmado`**
    (`contraparte_estado_vinculo === 'confirmado'` AND `contraparte_id != null`). A confirmed
    counterpart MAY render as a neutral internal link (future sub-maestra page); until that page
    exists, render plain text even when confirmed — never a dead link.
  - If the match is `probable` / `revision` / `no_confirmado` / `null` → render the raw string
    **plus `<IdentityMarker />`**, NEVER a link. Identical discipline to Phase 10 state (b).
  - **NEVER expose the counterpart's RUT.** Third parties are private individuals/firms; their
    RUT is internal-only (LEGAL-03, deny-by-default RLS) and must not be selected into the public
    payload, never shown, never put in a tooltip or `title`.
  - **No qualifier of the counterpart's role beyond what the source states.** Show the literal
    `tipo` if the source provides it ("lobbyist registrado" / "gestor de intereses") as plain
    metadata; never editorialize ("influyente", "vinculado a", "cercano a").
- **Asunto/materia (optional):** if the source publishes the subject of the audience, show it
  verbatim, `text-sm`. If absent, omit — never fabricate or summarize. No LLM-generated summary
  of what the meeting "was really about".
- **ProvenanceBadge:** per row, mandatory. `sourceName = "Ley del Lobby"` (or `sourceLabel(origen)`
  → "leylobby.gob.cl"), `capturedAt = fecha_captura`, `sourceUrl =` the per-audience source record
  link. If provenance is unknown → badge still renders with `sourceName="fuente desconocida"`,
  no link (§6.3). Never omit the badge.

**No CamaraChip per row.** The chamber/role lives in the header (`ParlamentarioHeader`); repeating
it per lobby row adds nothing and risks compositing identity context into the lobby datum. Keep
the row about: when, with whom (raw), about what (raw), per which source.

**Pagination:** if a legislator has many audiences, paginate server-driven via `?lobbyPage=N`
(page size **20 audiences**), with "Anteriores" / "Siguientes" anchors (`min-h-[44px]`) and
"Página N de M" — identical pattern to Phase 10 §3.2. SSR so a journalist can deep-link a page.
No infinite scroll.

### 3.3 Identity guard — meeting ↔ legislator (LOCKED, INT-01/02, riesgo #1)

The audience attaches to THIS profile only when the legislator (sujeto pasivo) matched
`determinista` / `confirmado` via `correrPipeline`, fixed through the typed `EnlaceConfirmado`
invariant (Phase 9). Each adjudication leaves a row in `identidad_audit`.

- A `confirmado` audience → renders in this section as a normal `LobbyAudienciaRow`, attributed
  to this legislator.
- A `probable` / `revision` / `no_confirmado` audience → its FK is NULL; it is **never silently
  attributed** to this profile and **never aggregated** into any count for this legislator. If
  surfaced at all it lives in a clearly-labeled "menciones sin verificar" area with the raw
  legislator mention + `<IdentityMarker />` (mirrors Phase 10 §3.6 state (b)) — but the default
  is simply to exclude it from this profile.
- This guard (legislator side) is **independent** of the counterpart guard (§3.2). Two separate
  identity decisions per row; neither is allowed to upgrade the other.

### 3.4 NO aggregate / NO viz / NO cross-composition (anti-feature, LOCKED)

This section is a **descriptive list only**. Explicitly PROHIBITED this phase (FEATURES.md INT
anti-features + CONTEXT lane rule):

- **No "lobby influence score"**, no meeting-frequency ranking, no "top counterparts" leaderboard,
  no heat/intensity coloring. A raw meeting *count* sentence ("N reuniones registradas") is
  permitted as neutral fact; anything that ranks, scores, or weights is not.
- **No CSS bar / no chart.** Unlike Phase 10 attendance, there is no honest aggregate to plot here
  yet; do not invent one. (`VotacionBar` exists but is not used.)
- **No conflict-of-interest flag/badge**, no "posible conflicto", no linking a meeting to a vote,
  a bill, a donation, or a declared interest. **No co-occurrence claim** ("se reunió con X antes
  de votar Y").
- **No composition across lanes:** a lobby row must not embed, reference, or sit inside a vote
  unit, and vice versa. Cross-block adjacency is deferred (CONTEXT "Deferred Ideas") and gated by
  legal review.

## 6. States

### 6.1 Empty / honest-empty states (LOCKED — never reads as "no lobby activity")

The existential distinction for this phase: **"no hemos ingerido" ≠ "no tiene reuniones".** A
legislator with zero ingested audiences must NOT look identical to one with a clean, fully-loaded
record of zero meetings. Three honest states, mirroring Phase 10 §3.6:

| Scope | Trigger | Copy |
|-------|---------|------|
| Lobby section — not yet ingested | confirmed-audience query returns 0 AND ingestion not yet run for this legislator | "Aún no hemos ingerido las reuniones de lobby de este parlamentario. Esto no significa que no se haya reunido — los datos de la Ley del Lobby se están incorporando." |
| Lobby section — ingested, zero confirmed | ingestion ran, 0 confirmed audiences for this legislator | "No se registran reuniones de lobby confirmadas para este parlamentario en el periodo consultado, según leylobby.gob.cl." |
| Counterpart unverified | meeting confirmed to legislator, but counterpart link not `confirmado` | render raw counterpart string + `<IdentityMarker />` (NOT an empty state — the meeting is real; only the counterpart identity is unverified) |
| Legislator mention unverified | source lists an audience whose legislator link is not `confirmado` | excluded from this profile by default; if surfaced, "menciones sin verificar" area with raw mention + `<IdentityMarker />` — never attributed, never counted |

Copy principles (v1.0 §6.1 / Phase 10, hardened): state the fact, give context, never say "error"
for missing data, never "Oops", **never let an empty read as virtue, cleanliness, or absence of
lobbying.** Always distinguish "no ingestado todavía" from "ingestado, sin resultados".

### 6.2 Loading states

shadcn `Skeleton`, shape-matched: `LobbySkeleton` mirrors the real structure — intro line skeleton
+ 3 audience-row skeletons (fecha bar + contraparte bar + provenance bar). One skeleton per
Suspense boundary; container `aria-hidden="true"` (no focus). Pattern identical to Phase 10
`VotosSkeleton`.

### 6.3 Error states

Inherited from v1.0 §6.3 / Phase 10. A Supabase/network failure renders inside
`border border-destructive/20 bg-destructive/5 rounded-lg p-4 text-sm`:

"Ocurrió un error al cargar las reuniones de lobby. Si el problema persiste, puedes consultar la
fuente directamente en [leylobby.gob.cl ↗]." + `<button>Reintentar</button>` (`text-primary underline`).

- Distinguish a real DB/network error (throw → honest error UI) from "no rows" (→ §6.1 empty).
  Use checked errors / `.maybeSingle()` discipline as in `/proyecto/[boletin]` — never mask a
  transient error as "sin reuniones".
- **Degraded ingestion (CONTEXT):** if `leylobby.gob.cl` is unreachable in the environment and the
  connector degraded to a fixture + `human_verification`, the section still renders only honest
  states — it never fabricates audiences. If a row is fixture-sourced, its ProvenanceBadge reflects
  that honestly (source/date as captured), never a fake official link.
- **Provenance unavailable:** render ProvenanceBadge with `sourceName="fuente desconocida"`, no
  link. Never hide the badge — its absence falsely implies the datum has no source.

### 9.1 Tone rules — HARD CONTENT RULE: anti-insinuation lane + anti-causality (riesgo existencial #2)

These are enforced at the component and copy level and are the most important constraint of the
phase. A checker/auditor failure on any of these BLOCKS the phase. They EXTEND Phase 10 §9.1
(which remains in force) with the multi-dataset lane discipline that Phase 11 establishes for the
whole parliamentary front.

1. **Lane isolation (the new LOCKED rule).** No UI unit composes a lobby meeting next to a vote
   (or any other dataset) as a single highlighted unit. Lobby lives in its own `<section id="lobby">`,
   its own `<h2>`, its own list, separated by the `mt-12` gutter. A lobby row never contains, links
   to, or visually pairs with a vote/bill/declaration. This pre-empts the "máquina de sospechas".
2. **No causality, ever.** Never assert or imply *why* a meeting happened or what followed from it.
   No "se reunió para", "a cambio de", "antes de votar", "que resultó en".
3. **No relationship/affinity language about the counterpart.** PROHIBITED: "cercano a", "vinculado a",
   "aliado de", "lobista habitual de", "su lobista", co-occurrence framing ("se reúne más seguido con").
   The counterpart is shown as the source lists it; the product never characterizes the relationship.
4. **No score / index / ranking / flag.** No lobby-influence score, no conflict-of-interest flag,
   no "frequency" ranking, no leaderboard of counterparts. Raw list (+ optional neutral count) only.
5. **No judgment adjectives** about a meeting or a counterpart ("polémico", "influyente", "oscuro",
   "controversial", "sospechoso").
6. **Counterpart identity uncertainty** uses exactly "identidad no verificada" — never a hedge.
7. **Third-party privacy is absolute.** Never expose a counterpart's RUT or any internal-only field;
   never imply wrongdoing by a third party. The counterpart is a private actor doing something legal
   and published.
8. **Provenance is mandatory** on every row; if unknown, "fuente desconocida" — never omitted.
9. **An empty is a fact, not a virtue.** Never phrase absence of ingested lobby data as cleanliness,
   integrity, or "no se reúne con nadie". "No ingestado" and "ingestado, sin reuniones" are distinct.

## 10. Provenance & Identity Data Flow (UI contract with backend)

Every lobby row carries:

```ts
interface LobbyAudienciaRow {
  audiencia_id: string;
  fecha: string;                       // ISO; audience date → DD MMM YYYY
  // legislator (sujeto pasivo) identity guard — §3.3
  parlamentario_id: string | null;     // set ONLY if confirmado
  parlamentario_estado_vinculo: "confirmado" | "probable" | "revision" | "no_confirmado" | null;
  parlamentario_mencion: string;       // raw legislator name from source (for the unverified case)
  // counterpart (lobbyist / firm) — §3.2, shown as RAW TEXT
  contraparte_nombre: string;          // RAW source string — ALWAYS shown verbatim
  contraparte_tipo: string | null;     // literal source role label, if any (no editorializing)
  contraparte_id: string | null;       // → lobby_contraparte sub-maestra, set ONLY if confirmado
  contraparte_estado_vinculo: "confirmado" | "probable" | "revision" | "no_confirmado" | null;
  // NOTE: contraparte RUT is NEVER part of the public payload (LEGAL-03 deny-by-default)
  asunto: string | null;               // verbatim subject of the audience, if published
  origen: string;                      // → sourceLabel() → ProvenanceBadge.sourceName ("Ley del Lobby")
  fecha_captura: string;               // → ProvenanceBadge.capturedAt
  enlace: string | null;               // → ProvenanceBadge.sourceUrl (per-audience source record)
}
```

- The `@obs/lobby` connector populates provenance per row and sets BOTH identity links via the
