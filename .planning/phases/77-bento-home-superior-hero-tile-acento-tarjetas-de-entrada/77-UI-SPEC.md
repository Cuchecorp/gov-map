### Cardinal rule (MILESTONE §0 + CONTEXT §specifics + CR-01): zero new hex, everything by token

The mockup is drawn ON the current palette. **No `#` hex literal enters any component this phase.** Every color references an existing CSS token. Tailwind 4 arbitrary syntax is **always** `[var(--token)]`, **never** `[--token]` (CR-01 from Phase 76: `[--token]` compiles to invalid CSS in Tailwind 4). The mockup hex → token map (verified by HSL conversion, Phase 76):

| Mockup hex (this phase's rows) | Existing token | Tailwind util |
|---|---|---|
| `#F9F6F0` (page + hero input fill) | `--background: 40 33% 97%` | `bg-background` |
| `#FDFBF7` (hero + entry tile fill) | `--card: 40 30% 99%` | `bg-card` |
| `#E3DDD3` (borders) | `--border: 40 16% 86%` | `border-border` |
| `#0F1729` (text) | `--foreground: 222 47% 11%` | `text-foreground` |
| `#5C6373` (secondary text, entry `→` glyph) | `--muted-foreground: 222 14% 42%` | `text-muted-foreground` |
| `#2A5859` (petróleo: accent-tile fill, button, marker, focus) | `--accent-product: 183 38% 26%` | `bg-accent-product` / `text-accent-product` |
| `#234A4B` (accent hover) | derived hover on `--accent-product` (see Accent Tile Hover) | NOT a new hex |
| `#EAF2F1` / `rgba(234,242,241,·)` (accent-tile foreground) | new `--accent-product-foreground` token (see Accent Foreground Fix) | `text-accent-product-foreground` |

### Accent Foreground Fix (resolves 76-UI-REVIEW warning #1 — dark AA)

The Phase 76 `accent` variant used `text-primary-foreground`, which inverts to near-black in dark and collapses contrast on the mid-teal dark fill. **This phase introduces a dedicated, theme-stable accent foreground token and fixes the accent fill in dark.**

Add to `globals.css`:

```css
:root {
  /* Bento accent tile — foreground legible sobre petróleo (AA ≥4.5:1).
   * Mapea #EAF2F1 del mockup a FULL alpha (no rgba .85). */
  --accent-product-foreground: 183 30% 96%;   /* ≈ #EAF2F1 */
}
.dark {
  /* En dark el accent-product base sube a 183 34% 46% (mid-teal) → texto blanco
   * cae a ~3.3:1. Para el TILE accent NO usamos ese mid-teal: la variante accent
   * fija el fill en el petróleo oscuro para preservar blanco AA. */
  --accent-product-foreground: 183 30% 96%;
}
```

Register in the `@theme inline` block (bare, no `hsl()` wrapper is wrong — these are HSL triples, so wrap as needed; follow the existing `--color-accent-product-soft` pattern, which uses a full-`hsl()` source → bare reference). Concretely, expose a Tailwind util `text-accent-product-foreground` / `bg-accent-product-foreground` consistent with how `--color-accent-product-soft` is wired (globals.css:77). **Executor: match the existing token-registration idiom exactly (gotcha 54-04: double-`hsl()` = invalid color).**

**Verified contrast (this phase's authority):**

| Pair | Ratio | Verdict |
|------|-------|---------|
| white `#FFFFFF` on `--accent-product` light (`#2A5859`) | 7.96:1 | PASS |
| `#EAF2F1` full-alpha on `#2A5859` | 7.00:1 | PASS (use this — full alpha) |
| `#EAF2F1` at mockup's `.85` alpha on `#2A5859` | 5.59:1 | passes, but use full alpha for margin + token-purity |
| white on **dark** `--accent-product` `183 34% 46%` | 3.30:1 | **FAIL** — do not use mid-teal as accent-tile fill in dark |
| `#EAF2F1` on dark mid-teal | 2.90:1 | **FAIL** |

**Contract:** accent-tile body/heading/CTA all use `text-accent-product-foreground` at **full alpha** (never `.85`). The accent-tile **fill stays petróleo `183 38% 26%` in BOTH themes** (do not let it lighten to the dark mid-teal), so white/near-white text stays ≥7:1. Implement by having the `accent` variant reference a fill that is dark-stable (e.g. a `--bento-accent-fill` fixed at `183 38% 26%`, or override the accent variant to not follow the `.dark` mid-teal). Full dark verification is Phase 80; this phase must not ship the failing pairing.

### Accent Tile Hover (resolves 76-UI-REVIEW warning #2)

The accent tile is a link (`<a href="/sobre">`). It needs a hover affordance the Phase 76 primitive lacked. **Contract:** `hover:bg-accent-product/90` (darkens toward mockup `#234A4B`) on the `accent` variant. This is a derived opacity of the token — no new hex. Add it to the `bentoTileVariants` `accent` string (`bg-accent-product ... hover:bg-accent-product/90`). Keep the existing `focus-visible` ring.

### Responsive collapse (resolves 76-UI-REVIEW warning #3)

`BentoGrid` is `grid-cols-1 md:grid-cols-6`; collapse to full width ≤`md` relies on each child being a `BentoTile` whose `span` maps to `md:col-span-N` (no base `col-span`). **Contract for this phase:** every child mounted here (hero, accent tile, 3 entry cards) is rendered **as a `BentoTile`** (via `asChild` wrapping a `<section>`/`<Link>`), so each self-declares its span and collapses correctly. Do NOT drop a raw `<section>`/`<a>` directly into `BentoGrid` — if unavoidable, it must carry `col-span-full md:col-span-N` itself. **Documented collapse order ≤md (DOM = visual):** hero → accent tile (`¿Cómo leer esto?`) → entry card 1 (Proyectos) → entry card 2 (Parlamentarios) → entry card 3 (Agenda). This DOM order is the collapse order; the primitive requires no reordering utilities.

**Dark pair:** `--card`/`--border`/`--background`/`--muted` all have dark values; the accent foreground + fixed accent fill above complete the accent tile's dark story. No other new dark tokens needed. Full dark verification: Phase 80.
