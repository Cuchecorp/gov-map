# Phase 61 Plan 02: Deploy #1 — Subir v6 (59+60+61-01) a producción

## Objective

Construir y desplegar el frontend a Cloudflare Workers con todo el trabajo de v6 incluido: Phase 59 (autoría), Phase 60 (brand), Phase 61-01 (leyendas cruces + triple-requisito charts). Deploy vía GitHub Actions (Linux, reproducible, sin problemas de symlinks Windows). Verificar que el Worker responde 200 con contenido esperado en las superficies clave.

## Context

- Runbook: `docs/deploy-cloudflare.md` — GitHub Actions es la ruta recomendada (Opción A).
- Workflow: `.github/workflows/deploy-cloudflare.yml` — `workflow_dispatch` manual.
- Worker URL: `https://observatorio-congreso.thevalis.workers.dev/`
- Gotchas de deploys anteriores (MEMORY): Docker Linux si fallara CI; `MSYS_NO_PATHCONV=1` para paths en bash; `docker cp` con ruta Windows explícita; nunca build OpenNext en Windows directo (EPERM symlinks).
- Secrets CI ya configurados: `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID`.
- Runtime secrets del Worker ya seteados (deploys anteriores): `SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `GEMINI_API_KEY`, `CRUCES_PUBLIC_ENABLED`.

## Tasks

### Task 1 — Verificar working tree limpio y pushear master

**type:** auto

Push todos los commits de 61-01 (y fases anteriores) a origin/master para que GitHub Actions los vea.

**done criteria:**
- `git status` limpio.
- `git push origin master` exitoso.
- `git log origin/master..HEAD` vacío (no hay commits locales pendientes).

### Task 2 — Disparar deploy vía GitHub Actions

**type:** auto

Disparar el workflow `deploy-cloudflare` vía `gh workflow run deploy-cloudflare.yml`. Esperar a que termine (máx 10 min). Capturar el run ID y el resultado (success/failure).

**done criteria:**
- `gh run list --workflow=deploy-cloudflare.yml --limit=1` muestra el run más reciente en estado `completed` / `success`.
- No hay errores de build en los logs.

### Task 3 — Capturar versión de deploy y URL

**type:** auto

Extraer de los logs del run: deployment version ID (Cloudflare) y URL del Worker. Verificar con curl que el Worker responde 200.

**done criteria:**
- `curl -s -o /dev/null -w "%{http_code}" https://observatorio-congreso.thevalis.workers.dev/` = 200.
- Version ID anotada para el SUMMARY.

### Task 4 — Verificación LIVE de superficies clave

**type:** auto

Verificar con curl/HTTP las siguientes superficies:

| Check | Comando | Esperado |
|-------|---------|---------|
| Home (200 + brand) | `curl -s https://observatorio-congreso.thevalis.workers.dev/` | HTTP 200, body contiene "gov-map" o svg del lockup |
| /icon.svg | `curl -s -o /dev/null -w "%{http_code}" .../icon.svg` | 200 |
| /buscar | `curl -s -o /dev/null -w "%{http_code}" .../buscar` | 200 |
| Ficha parlamentario (datos ricos) | `curl -s -o /dev/null -w "%{http_code}" .../parlamentario/[id]` | 200 |
| Ficha proyecto moción (con autores) | `curl -s -o /dev/null -w "%{http_code}" .../proyecto/[boletin]` | 200 |

Para encontrar IDs/boletines con datos, leer `.env` si tiene `DATABASE_URL` o usar `SUPABASE_URL`+`SUPABASE_SECRET_KEY` para un read-only query directo.

**done criteria:**
- Todas las superficies responden 200.
- Home contiene indicador del nuevo brand/lockup.
