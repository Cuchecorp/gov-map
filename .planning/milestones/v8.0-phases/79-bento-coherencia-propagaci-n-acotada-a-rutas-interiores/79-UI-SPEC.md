# Phase 79 — UI Design Contract

> Visual and interaction contract for BENTO-COHERENCIA — bounded propagation of the bento chrome (1120px container + `--radius-tile` first-level corners) to interior routes, **without re-laying-out any interior page**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note (D3 — RESOLVED, autonomous):** this phase is a **surgical swap**, not a redesign. Per route it does exactly TWO things: (1) widen the page's own `<main>` container from `max-w-3xl`/`max-w-5xl` → `max-w-[1120px]`, and (2) swap the corner radius of **first-level cards only** from `rounded-lg`/`rounded-md` → `rounded-[var(--radius-tile)]`. **Interiors are byte-identical** (accordions, charts, tables, skeletons, sub-rows). **Zero re-layout, zero new hex, zero data/RPC change.** `/red` is **EXCLUDED** (documented, invariant 4). `/admin/revisar-entidades` is **not touched**.

### Cardinal rule (MILESTONE §0 + CONTEXT §Discretion + invariant 2): zero new hex, everything by token

No `#` hex literal enters any file this phase. The only new class introduced is `rounded-[var(--radius-tile)]` (the token minted in Phase 76). Tailwind 4 arbitrary syntax is **always** `[var(--radius-tile)]`, **never** `[--radius-tile]` (CR-01, Phase 76: `[--token]` compiles to invalid CSS in Tailwind 4). No color, spacing, typography, or layout token changes — this phase reads the container/radius contract already established in 76/77 and applies it to interior `page.tsx` files only.

### The `Card` primitive must NOT be edited (D3 firewall)

`app/components/ui/card.tsx` line 12 hardcodes `rounded-lg`. **Do NOT change it.** `Card` is consumed both by first-level cards (`SearchResultCard`) AND by interior/nested panels and skeletons across every route. Editing the primitive would radius-swap interiors too → **D3 violation** (interiors must stay byte-identical). The swap is applied **at the first-level call site** via a `className` override that `cn()`-merges over the primitive's `rounded-lg` (tailwind-merge wins last-writer), or by swapping the literal `rounded-lg`/`rounded-md` on a hand-rolled first-level card `<div>`/`<li>`/`<Link>`. Enumerated per route below.

## Sticky-Header / `scroll-mt` Interaction (mandatory fix — fichas only)

**Verified conflict.** Phase 76 added a **global** `scroll-margin-top: 5rem` (=80px, `scroll-mt-20`) in `globals.css:106` so the sticky header does not occlude anchor targets. The ficha sections still carry a **local** `scroll-mt-6` (=24px) on each `<section id=…>` (parlamentario L271-459, proyecto L102-201). In Tailwind, the later/more-specific utility `scroll-mt-6` **wins over** the global rule → an anchor jump lands the section **24px** below the viewport top, and the **80px sticky header covers it** (56px of the heading is hidden behind the header).

**Contract (executor MUST apply):** on the ficha routes only, the per-`<section>` local `scroll-mt-6` must be reconciled with the global 80px header offset. Options, in order of preference:
1. **Remove the local `scroll-mt-6`** from the ficha `<section id=…>` elements so the global `scroll-margin-top: 5rem` applies uniformly (cleanest — anchors clear the header). Confirm the FichaRail scrollspy still highlights correctly.
2. If a section needs extra breathing room beyond 80px, raise the local to `scroll-mt-24` (96px) — never leave it at `scroll-mt-6` (24px), which is provably occluded.

**Not the same as `md:top-6`.** `FichaRail` is `md:sticky md:top-6` (`ficha-rail.tsx:48`) — that positions the **rail column**, not the anchor offset. Leave `md:top-6` unchanged unless the visual capture shows the rail itself tucking under the header (if so, raise to `md:top-20` to match the header). This is a rail-position concern distinct from the anchor `scroll-mt` fix above.

**Test:** `scroll-mt-6` on ficha sections should no longer be provably occluded; the fix is verifiable in the Phase 81 deploy-real anchor-jump capture. jsdom cannot see this (no layout) — it is a visual/BrowserOS check.
